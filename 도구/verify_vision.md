# Vision 검증 절차 (Claude / GPT-4V 등)

DOM 기반 검증(Playwright)이 잡지 못하는 **시각적 의도**를 검증하는 절차. AI 에이전트가 자동으로 수행할 수 있도록 명세화.

---

## 언제 쓰나

| 검증 항목 | 도구 |
|---|---|
| 타이밍 정확도 (요소가 t초에 보이는가) | Playwright DOM 검사 |
| 시각적 무드·가독성·정보 과밀 | **Vision LLM** |
| 컬러 대비, 자막 가독성, 의도된 강조 | Vision LLM |
| 발화 키워드와 화면 키워드 일치 | Vision LLM (자막 OCR 같이 활용) |

---

## 워크플로우 (에이전트가 따라 하는 단계)

### 1. 검증 대상 시점 결정

`sync_map.json` 의 `resolved` 시점들 + 각 씬의 중간 지점을 합쳐 시점 리스트를 만든다.

```bash
# 예: 001 프로젝트
cat 06_편집/검증/sync_map.json | jq '.resolved | values | sort'
```

추가로 각 씬의 (start+end)/2 시점을 더한다 (씬 무드 검증용).

### 2. 스크린샷 캡처

```bash
cd 06_편집/hyperframes
npx --yes hyperframes@0.6.7 snapshot --at <쉼표 구분 시점 리스트>
# 결과: snapshots/frame-00-at-XX.Xs.png ...
```

### 3. 의도 정보 로딩

다음 3개 파일을 함께 읽는다:
- `03_스토리보드/design.md` — 비주얼 컨셉·씬별 디렉션·자동 품질 체크리스트
- `03_스토리보드/스토리보드.md` — 씬별 화면 레이아웃 명세
- `06_편집/검증/sync_map.json` — 해석된 시점

### 4. Vision LLM 평가 (씬별 + 시점별)

각 스크린샷에 대해 다음 질문을 LLM에게 묻는다:

#### 4-1. 무드·컨셉 일치도
> "이 프레임은 design.md의 비주얼 컨셉 '{컨셉 한 줄}'에 부합합니까? 차분/임팩트/친근 같은 무드 키워드가 살아 있나요? 1-5점, 사유 한 줄."

#### 4-2. 정보 과밀도
> "한 화면에 텍스트 라인이 6개를 초과합니까? 시청자가 1.5초 안에 스캔 가능한 양인가요?"

#### 4-3. 발화-화면 키워드 일치
> "이 시점의 자막은 '{자막 텍스트}'입니다. 화면에 자막과 호응하는 시각 강조(같은 키워드의 타이틀·아이콘 등)가 있습니까?"

#### 4-4. 자막 가독성
> "자막 박스 대비, 글자 크기, 위치가 가독성 기준에 맞습니까? 다른 시각 요소에 가려지지 않습니까?"

#### 4-5. 의도된 강조 유무
> "design.md의 씬 디렉션에 따르면 이 시점에 '{핵심 비주얼}'이 강조되어야 합니다. 실제 프레임에서 그 요소가 시선의 중심에 있습니까?"

### 5. 종합 리포트 작성

`06_편집/검증/verify_vision_report.md` 에 다음 형식으로 작성:

```markdown
# Vision Verify Report

- 생성 시각: YYYY-MM-DD HH:MM
- 검증 프레임: 12개 (시점 + 씬 중간)

## 종합
- 무드 일치: 평균 4.3 / 5.0
- 정보 과밀 위반: 0건
- 발화-화면 키워드 불일치: 1건 (S04 38.0s "Strategic Tech Trends 2025" 글자가 너무 작음)
- 자막 가독성 이슈: 0건
- 의도된 강조 누락: 1건 (S02 16.0s 우측 파이프라인이 시선 중심 아님)

## 시점별 상세
### 4.5s (S01)
- 무드: 5/5 ("호기심·활기" 살아 있음)
- 자막: "여러분이 AI에게 일을 시켰는데," — 가독성 OK
- 강조: ✓ 마크가 중앙 명확, 의도 부합 ✅
- 개선 제안: 없음

### 16.0s (S02)
...
```

### 6. FIX 단계로 피드백

`verify_vision_report.md` 에서 FAIL/개선 제안이 있으면:
1. `design.md`의 의도가 명확한가? (애매하면 명세 보강)
2. GSAP 시점/스타일이 의도와 맞는가? (코드 수정)
3. 수정 후 다시 1단계로

## 에이전트 호출 패턴 예시

```
사용자: "001 프로젝트 vision 검증 돌려줘"

에이전트:
1. cat 06_편집/검증/sync_map.json | jq '.resolved' 로 시점 추출
2. 각 씬 중간 시점 6개 추가
3. hyperframes snapshot --at <시점 리스트>
4. Read tool로 design.md, 스토리보드.md, sync_map.json 로딩
5. 각 스크린샷 PNG를 Read tool로 읽음 (Claude의 멀티모달 입력)
6. 위 4-1~4-5 질문을 순서대로 평가
7. verify_vision_report.md 작성
8. FAIL/개선 제안 요약을 사용자에게 보고
```

## DOM 검증과의 분담

| 항목 | Playwright (DOM) | Vision LLM |
|---|---|---|
| 타이밍 정확성 (±0.15s) | ✅ 정확 | ⚠️ 부정확 |
| 요소 가시성 (opacity·display) | ✅ 정확 | ✅ 직관적 |
| 컬러·대비·여백 | ❌ | ✅ |
| 정보 과밀·시선 흐름 | ❌ | ✅ |
| 의도된 무드 | ❌ | ✅ |
| 화면 텍스트 OCR 일치 | ⚠️ DOM에서 추출 가능 | ✅ |
| 출력 형식 | 자동 PASS/FAIL | 자연어 평가 |

→ **두 검증은 보완 관계**. Playwright로 빠르게 1차 필터링 → Vision으로 미묘한 디자인 이슈 잡기.
